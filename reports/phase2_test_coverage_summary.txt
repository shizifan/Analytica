━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 业务化程度不足
   ├─ 问题: 测试输入过于抽象（"测试指标" vs "集装箱TEU完成率"）
   ├─ 影响范围: test_planning.py, test_state_machine.py 全部11个TC
   ├─ 风险: 无法验证真实用户意图→LLM学习能力无法评估
   └─ 与Phase1优化文档的同步问题: ⚠️ 相同问题再现

2. 端点覆盖不完整
   ├─ 当前: 15/27 端点 = 56%覆盖率
   ├─ 缺失: 6个端点 (M06,M07,M08,M09,M16,M21)
   │        ├─ M06: getVesselEfficiency (船舶作业效率)
   │        ├─ M07: getPortInventory (港存库容)
   │        ├─ M08: getDailyProductionDynamic (日度动态)
   │        ├─ M09: getShipStatus (船舶状态)
   │        ├─ M16: getMarketBusinessSegment (市场板块)
   │        └─ M21: getCustomerBasicInfo (客户基础)
   ├─ 影响: LLM可能生成这6个端点的幻觉
   └─ 风险等级: 🔴 高 (22% 盲点)

3. 关键代码路径未覆盖
   ├─ PlanningEngine._build_prompt() → ❌ 完全未测
   │   └─ domain-aware endpoint filtering 的逻辑
   ├─ PlanningEngine._validate_tasks() → ⚠️ 部分测
   │   └─ M-code多格式解析 (M01, M1, m01等)未测
   ├─ update_plan() → ⚠️ 仅测remove_task
   │   └─ modify_task, add_task 未测
   ├─ _break_cycles() → ⚠️ 1个路径
   │   └─ 复杂DAG的循环修复未全测
   └─ 影响: 规划引擎的核心业务逻辑无法验证

4. 状态机转移路径不完整
   ├─ 已测: perception→planning (2条), planning→END/execution (2条)
   ├─ 缺失:
   │   ├─ execution → planning (replan循环)   ❌
   │   ├─ execution → reflection (all done)   ❌
   │   ├─ execution → execution (loop)        ❌
   │   ├─ 异常路径 (timeout, error state)     ❌
   │   └─ 并发冲突 (同session多请求)         ❌
   └─ 风险: 错误恢复、重新规划逻辑未验证

5. E2E端到端串联完全缺失
   ├─ 缺失完整流程: 用户输入 → 感知 → 规划 → 修改 → 确认 → 执行
   ├─ 当前只有单点测试，无跨层集成
   ├─ 无法检测:
   │   ├─ 感知输出 → 规划输入的适配性
   │   ├─ 状态机各节点间的数据流转
   │   └─ 版本控制、修改日志在整个流程中的一致性
   └─ 风险: 关键业务流程的bug无法及时发现

6. 集成测试薄弱
   ├─ 已测: GET plan (3个), POST confirm (2个) = 基础CRUD
   ├─ 完全缺失:
   │   ├─ POST /plan/regenerate (用户修改重新规划)     🔴 高
   │   ├─ POST /plan/rollback (版本回退)              🔴 高
   │   ├─ GET /plan/versions (版本历史)               🟡 中
   │   └─ 并发请求处理、网络中断恢复                 🟠 低
   └─ 影响: 用户核心功能路径 (regenerate/rollback) 未测
